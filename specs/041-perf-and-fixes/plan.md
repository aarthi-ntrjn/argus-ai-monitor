# Implementation Plan: 041-perf-and-fixes

## Context

This branch addresses PTY session reliability, UX polish, and a new remote URL rescan feature.
All changes are incremental improvements to existing subsystems.

## Architecture Decisions

### PTY Reconnect via `ptyLaunchId`

The `pty_launch_id` column was added to the `sessions` table so the server can
re-link a reconnecting launcher to its existing DB session without waiting for
the next scan cycle. The `ptyLaunchId` is generated by `launch.ts` as a UUID
and passed via `?id=` in the WebSocket URL.

### Session Selection Persistence

`selectedSessionId` in `DashboardPage` is initialised from `localStorage` and
written back on every change. The close button clears the value. This keeps
the implementation local to the dashboard page with no new services.

### Rescan Remote URLs Endpoint

`POST /api/v1/repositories/rescan-remotes` runs `git remote get-url origin` for
all repos in parallel and updates only those that changed. Broadcasts
`repository.updated` for each changed repo so the frontend refreshes live via
the existing WebSocket invalidation path.

## Implementation Phases

### Phase 1 — PTY Reliability
- Persist `pty_launch_id` in DB
- Generate `ptyLaunchId` in `launch.ts` and include in WS URL
- Immediate reconnect on `register` message when existing session found
- Fix `notifySessionEnded` to include `sessionId`
- Fix `claimByTempId` → `claimByPtyLaunchId` rename in tests

### Phase 2 — UX Improvements
- Session card: neutral hover, blue selection
- Output pane: single-click opens, X-only closes
- Output pane: selection state persisted in localStorage
- Settings panel: wider padding, narrower width

### Phase 3 — Rescan Remote URLs
- Backend: `POST /api/v1/repositories/rescan-remotes`
- Frontend: `rescanRemoteUrls()` in `api.ts`
