# Implementation Plan: Supply Chain Attack Protection

**Branch**: `010-supply-chain-hardening` | **Date**: 2026-04-03 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/010-supply-chain-hardening/spec.md`

## Summary

Protect the Argus repository against supply chain attacks by: (1) creating a GitHub Actions CI pipeline that enforces lockfile-only dependency installs; (2) pinning all third-party action references to immutable commit SHAs with automated validation; (3) suppressing npm lifecycle scripts by default with an explicit allowlist for necessary exceptions; and (4) adding a PR-time dependency review step that blocks merges introducing packages with critical or malicious advisories.

The project has no existing GitHub Actions workflows — this feature creates the CI infrastructure from scratch.

### Architecture Decisions

**AD-001: Two-workflow split**
`ci.yml` runs on every push and PR (build + test + audit + pin validation). `supply-chain.yml` runs only on PRs (dependency review). This avoids running the dependency diff check on pushes where there is no base to compare against.

**AD-002: `npm ci --ignore-scripts` as default install command**
Every `npm ci` invocation in CI uses `--ignore-scripts`. The allowlist file drives a follow-up `npm rebuild` step. This is the most transparent and maintainable approach — the allowlist is a single source of truth checked into the repo and visible in PR diffs.

**AD-003: Custom shell script for pin validation**
`validate-action-pins.sh` runs *inside* the same `ci.yml` workflow it is validating. This means the workflow itself must be correctly pinned for the script to run — bootstrapping integrity. New workflow files are checked on their first CI run.

**AD-004: `dependency-review-action` fail-on-severity: critical**
The action is configured with `fail-on-severity: critical` and `deny-licenses: []` (no license blocking). This satisfies FR-008 without producing false positives from moderate/low advisories that affect only development dependencies.

**AD-005: Node.js version pin**
The CI workflow pins the Node.js version (matching the version used locally) to avoid version-drift supply chain risk. The version is set explicitly in the workflow rather than relying on an `.nvmrc` file.

### Workflow Designs

#### `ci.yml` — Step sequence

```
trigger: push (all branches), pull_request (→ master)

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      1. actions/checkout@<SHA>          # pin to immutable SHA
      2. actions/setup-node@<SHA>        # pin to immutable SHA
         with: node-version: '22'
      3. npm ci --ignore-scripts         # lockfile enforcement + script suppression
      4. bash .github/scripts/validate-action-pins.sh   # pin validation
      5. npm rebuild <allowlisted pkgs>  # selective rebuild for native modules
      6. npm test --workspace=backend    # vitest
      7. npm run build --workspace=frontend  # tsc + vite build
      8. npm audit --audit-level=critical    # known CVE check
```

#### `supply-chain.yml` — Step sequence

```
trigger: pull_request (→ master)

jobs:
  dependency-review:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write
    steps:
      1. actions/checkout@<SHA>
      2. github/dependency-review-action@<SHA>
         with: fail-on-severity: critical
```

### `validate-action-pins.sh` — Logic

```sh
#!/usr/bin/env bash
# Fail if any third-party GitHub Action uses: directive lacks a full SHA pin.
# Local actions (./...) are exempt.

ERRORS=0
while IFS= read -r line; do
  file=$(echo "$line" | cut -d: -f1)
  lineno=$(echo "$line" | cut -d: -f2)
  uses=$(echo "$line" | grep -oP 'uses:\s*\K\S+')
  # Skip local actions
  [[ "$uses" == ./* ]] && continue
  # Check for 40-char hex SHA after @
  if ! echo "$uses" | grep -qP '@[0-9a-f]{40}'; then
    echo "UNPINNED: $file:$lineno  $uses"
    ERRORS=$((ERRORS + 1))
  fi
done < <(grep -rn 'uses:' .github/workflows/*.yml)

exit $ERRORS
```

The following action SHAs must be resolved from GitHub releases before implementing the workflow files:

| Action | Current latest tag | SHA to verify at implementation |
|--------|--------------------|--------------------------------|
| `actions/checkout` | `v4` | Resolve from https://github.com/actions/checkout/releases |
| `actions/setup-node` | `v4` | Resolve from https://github.com/actions/setup-node/releases |
| `github/dependency-review-action` | `v4` | Resolve from https://github.com/github/dependency-review-action/releases |

**How to resolve**: `git ls-remote https://github.com/<owner>/<repo> refs/tags/<tag>` returns the tag object SHA; follow with `git cat-file -p <tag-sha>` to get the commit SHA, or use `gh release view <tag> --repo <owner>/<repo> --json targetCommitish`.
